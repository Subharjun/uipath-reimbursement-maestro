# ReimbursementClassificationAgent — Stage 3 (Subharjun)

AgentHack Track 1 · Reimbursement Process Automation · **Stage 3: AI Classify & Validate**

A UiPath **coded agent** (Python + LangGraph) that takes the expense data extracted
upstream (Stage 1 intake + Stage 2 IDP) and returns a structured classification:
expense type, business-purpose validity, duplicate flag, and a Low/Medium/High risk
score with audit trail. Hands off to Mir's Stage 4 (Policy Check).

## Architecture

```
START → classify → score_risk → END
```

| Node | Type | Responsibility |
|------|------|----------------|
| `classify` | LLM (`UiPathChat` gpt-4o, temp 0) **+ deterministic fallback** | expense_type, business_purpose_valid, classification_confidence |
| `score_risk` | Pure Python (deterministic) | risk_score, risk_factors, duplicate override, audit_entry |

- **LLM path** routes through the UiPath **LLM Gateway** (no API keys; uses Agent Units).
- **Fallback path** uses keyword matching from `data/mock_policy.json` — so the agent
  still runs and is fully testable locally without auth or units.
- Risk scoring is **always deterministic** (auditable, reproducible) regardless of path.

## Input contract (from Stage 2)

| Field | Type | Notes |
|-------|------|-------|
| `case_id` | string | required — Maestro case ID |
| `source_email_body` | string | raw email |
| `vendor` | string\|null | IDP-extracted |
| `date` | string\|null | YYYY-MM-DD |
| `amount` | number | may be 0 |
| `currency` | string | default INR |
| `document_attached` | bool | IDP confidence > 0.85 |
| `ocr_confidence` | number\|null | 0..1 |
| `employee_email` | string | sender |
| `duplicate_detected` | bool | set by upstream Orchestrator duplicate check; honoured as a High-risk override |

## Output contract (to Stage 4 — Mir)

This output is a **drop-in for Mir's Stage-4 agent input** — it contains every field
his `ReimbursementMainAgent` expects (pass-through + classification), plus audit extras
Mir can ignore. Null `vendor`/`date`/`reason` are defaulted to `"not provided"` so his
required inputs are always populated.

```json
{
  "case_id": "REIMB-2026-...",
  "expense_type": "travel|food|medical|internet|equipment|others",
  "amount": 7000,
  "currency": "INR",
  "vendor": "IndiGo Airlines",
  "date": "2026-03-11",
  "reason": "client meeting in Delhi",
  "document_attached": false,
  "risk_score": "Low|Medium|High",
  "duplicate_detected": false,
  "business_purpose_valid": true,
  "risk_factors": ["travel_high_amount_no_document"],
  "classification_confidence": "High|Medium|Low",
  "audit_entry": { "timestamp": "...", "stage": "classification", "stage_number": 3, "actor": "ReimbursementClassificationAgent", "actor_type": "agent", "action": "classification_complete", "details": "..." }
}
```

## Policy DB — `data/mock_policy.json`

Single source of truth for the whole team (Day-1 deliverable). Holds per-category
spend limits + auto-approve thresholds, the risk thresholds used by `score_risk`,
and the keyword lists used for classification/fallback. The repo-root copy
`../data/mock_policy.json` is the shared team copy — keep them in sync.

Risk rules implemented (all evaluated; highest wins):

- **High** — medical w/o document · travel > 5000 w/o document · type=others ·
  invalid purpose & amount > 1000 · suspicious phrase ("urgent approval"/"ignore"/"bypass") ·
  duplicate detected.
- **Medium** — no document & amount > 1000 · travel 2000–5000 · OCR < 0.85 · missing date.
- **Low** — none of the above.

## Run it

```bash
source .venv/bin/activate

# single run (LLM if logged in, else deterministic fallback)
uip codedagent run agent '{"case_id":"CLASS-TEST-001","source_email_body":"travel to Delhi for client meeting","vendor":"IndiGo","amount":7000,"document_attached":false,"employee_email":"a@b.com"}' --output-file out.json

# smoke evaluation (4 cases, output-based)
uip codedagent eval agent evaluations/eval-sets/smoke-test.json --no-report
```

## Deploy

```bash
uip codedagent deploy --my-workspace   # or --tenant / --folder "<name>"
```

Then it appears in Orchestrator → Agents and can be invoked by the Maestro Case.
